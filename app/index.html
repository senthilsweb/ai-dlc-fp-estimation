How FP Types & Complexity Were Assigned

This guide explains the methodology used to assign Function Point types and complexity ratings to each capability in the WBS. We follow the IFPUG (International Function Point Users Group) standard counting practices manual (CPM).

Function Point Types

EI - External Input

Data or control information that enters the system from outside its boundary. Examples: form submissions, file uploads, API requests that modify data.

EO - External Output

Data that leaves the system with processing logic (calculations, transformations, derivations). Examples: reports with totals, dashboard metrics, export with aggregations.

EQ - External Inquiry

Simple data retrieval without significant processing. Examples: search results, detail views, lookup operations.

ILF - Internal Logical File

Logically related groups of data maintained within the system. Examples: database tables, configuration stores, user profiles.

EIF - External Interface File

Data referenced from external systems but not maintained by this application. Examples: external API data, third-party integrations.

Complexity Rating Methodology

Complexity is determined by counting Data Element Types (DETs) and related files:

Transactional Functions (EI, EO, EQ)

DETs 1-4 FTRs 5-15 FTRs 16+ FTRs
1-4 DETs Low Low Average
5-15 DETs Low Average High
16+ DETs Average High High

DET = Data Element Type (unique user-recognizable fields)
FTR = File Type Referenced (logical files accessed by the transaction)

Data Functions (ILF, EIF)

DETs 1 RET 2-5 RETs 6+ RETs
1-19 DETs Low Low Average
20-50 DETs Low Average High
51+ DETs Average High High

RET = Record Element Type (user-recognizable subgroups within the data function)

Assignment Examples from This Project

User Login Form → EI (Average)

Why EI: Data enters the system (username, password)
Why Average: 5-10 DETs (username, password, remember me, CAPTCHA, etc.), references 1-2 files (Users, Sessions) = Average complexity

Dashboard Metrics → EO (High)

Why EO: Data leaves with processing (calculations, aggregations)
Why High: 20+ DETs (multiple KPIs, charts, summaries), references 5+ files (Studies, Subjects, Sites, etc.) = High complexity

Study List View → EQ (Low)

Why EQ: Simple data retrieval, no complex processing
Why Low: 4-5 DETs (study name, status, date), references 1 file (Studies) = Low complexity

Study Configuration Store → ILF (Average)

Why ILF: Data maintained by the application
Why Average: 20-40 DETs (study metadata fields), 2-4 RETs (study, arms, visits, forms) = Average complexity

Glossary

IFPUG Function Point terms, the estimation model, and AI-DLC vocabulary.

What this tool is for. Estimating software delivery across the spectrum from human-led, through AI-assisted, to agent-led — where an AI agent does the bulk of the work and the human steers, reviews, decides and approves. AI-DLC is the reference methodology it is modelled on. Function Points remain the invariant measure — a function’s size does not change because an agent wrote it — which is precisely what makes agent-led and human-led delivery comparable. The delivery rate is what changes, and that is expressed through the configurable knobs in Effort & Cost. Published estimation benchmarks all assume a human writes the code; no calibrated model exists for agent-led delivery yet. Treat outputs as tunable projections until calibrated against your own actuals.
